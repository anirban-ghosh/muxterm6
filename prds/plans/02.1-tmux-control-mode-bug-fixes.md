## 2. Bug #1: Always Opens With 2 Tabs

**Bug description:** Every tmux window opened with 2 duplicate tabs instead of 1.

**How it manifested:** Two identical tabs appeared immediately upon `tmux -CC` attach.

**Root cause:** `React.StrictMode` double-mounts components in dev mode, calling `createNewTab()` twice in the `useEffect` that runs on component mount.

**Fix:** Added a `mountedRef` guard in `App.tsx` to skip the second mount.

**File modified:** `src/renderer/App.tsx`

```typescript
// BEFORE (broken):
const tmuxSessionIdRef = useRef<string | null>(null)

const createNewTab = useCallback(async () => { /* ... */ })

useEffect(() => {
  const sessionId = getTmuxSessionId()
  if (sessionId) {
    tmuxSessionIdRef.current = sessionId
  } else {
    createNewTab()
  }
}, [])

// AFTER (fixed):
const tmuxSessionIdRef = useRef<string | null>(null)
const mountedRef = useRef(false)

useEffect(() => {
  if (mountedRef.current) return // Guard against StrictMode double-mount
  mountedRef.current = true

  const sessionId = getTmuxSessionId()
  if (sessionId) {
    tmuxSessionIdRef.current = sessionId
  } else {
    createNewTab()
  }
}, [])
```

## 3. Bug #2: Window Not Movable

**Bug description:** The tmux window could not be dragged/moved by clicking the titlebar.

**How it manifested:** Clicking and dragging the titlebar area did nothing. The window was stuck in place.

**Root cause:** The `.tabbar` CSS had `-webkit-app-region: no-drag` covering the entire titlebar, leaving only the 78px traffic light spacer as draggable. Also, the `loadFile` search param in window creation had an extra `?` prefix that Electron didn't expect.

**Fix (two parts):**
1. Changed `.tabbar` to `drag`, added explicit `no-drag` on `.tab` items and `.tabbar__new` button
2. Fixed `loadFile` search param (removed extra `?` prefix)

**Files modified:** `src/renderer/global.css`, `src/main/window-manager.ts`

```typescript
// BEFORE (broken) — window-manager.ts:
const query = `?tmux=${encodeURIComponent(sessionId)}`
if (process.env.ELECTRON_RENDERER_URL) {
  win.loadURL(process.env.ELECTRON_RENDERER_URL + query)
} else {
  win.loadFile(join(__dirname, '../renderer/index.html'), { search: query })
}

// AFTER (fixed):
const queryString = `tmux=${encodeURIComponent(sessionId)}`
if (process.env.ELECTRON_RENDERER_URL) {
  win.loadURL(`${process.env.ELECTRON_RENDERER_URL}?${queryString}`)
} else {
  win.loadFile(join(__dirname, '../renderer/index.html'), { search: queryString })
}
```

## 4. Bug #3: Stdin Restricted to First Column

**Bug description:** All keyboard input in tmux panes was restricted to the first column. Typing appeared only at column 1, with text wrapping immediately.

**How it manifested:** Any text typed in the tmux pane wrapped after a single character.

**Root cause:** `refresh-client -C cols,rows` was never sent during initialization, so tmux used the control PTY's default size (likely tiny or matching the trigger terminal's dimensions). Panes were effectively 0 or 1 column wide.

**Fix:**
1. Added `refresh-client -C 200x50` during init as a reasonable default
2. Added resize reporting from TerminalView so tmux gets actual terminal size

**File modified:** `src/main/tmux/tmux-session.ts`, `src/renderer/components/Terminal/TerminalView.tsx`

```typescript
// Added to tmux-session.ts initialize():
await this.queue.send('refresh-client -C 200x50').catch(() => {})

// TerminalView.tsx handleResize:
if (tmuxPaneIdRef.current) {
  window.terminalAPI.resizeTmux(cols, rows)
} else {
  window.terminalAPI.resizePty(ptyIdRef.current, cols, rows)
  updateTerminalSize(ptyIdRef.current, cols, rows)
}
```

> **Note:** This was later found to be caused by a second root cause — see Bug #8 (comma vs x separator).

## 5. Bug #4: Detach Freezes Triggering Window

**Bug description:** After detaching from a tmux session (pressing Esc), the original triggering terminal window froze. No further input or output worked.

**How it manifested:** After `%exit` notification from tmux, the triggering terminal became unresponsive.

**Root cause (initial):** After detach, the PTY stayed in `tmuxMode=true`, so all subsequent output (shell prompt, `[detached]` message) went to the disposed TmuxSession instead of being rendered in the triggering terminal.

**Root cause (deeper, found in second debugging round):** After `%exit`, the parser continued processing remaining data in the buffer (shell prompt, `[detached]` message) as protocol lines, silently dropping them. The shell prompt was consumed by the parser.

**Fix (two rounds):**

**Round 1 — Clear tmux mode on session end:**

```typescript
// Added to pty-manager.ts:
clearTmuxMode(ptyId: string): void {
  const instance = this.ptys.get(ptyId)
  if (instance) {
    instance.tmuxMode = false
    instance.dcsBuffer = undefined
    logger.info({ ptyId }, 'pty tmux mode cleared')
  }
}

// tmux-manager.ts removeSession() now calls:
ptyManager.clearTmuxMode(ptyId)

// tmux-manager.ts createSession() adds onExit callback:
session.onExit = () => {
  this.sessions.delete(ptyId)
  ptyManager.clearTmuxMode(ptyId)
  logger.info({ ptyId }, 'tmux session ended via %exit')
}
```

**Round 2 — Parser returns remaining raw data after `%exit`:**

```typescript
// BEFORE — tmux-protocol-parser.ts:
feed(data: string): void {
  this.buffer += data
  this.processLines()
}

// AFTER:
private exited = false

feed(data: string): string | null {
  if (this.exited) {
    // After %exit, all data is raw shell output
    return data
  }
  this.buffer += data
  this.processLines()
  if (this.exited && this.buffer.length > 0) {
    const remaining = this.buffer
    this.buffer = ''
    return remaining
  }
  return null
}

// pty-manager.ts now checks return value:
if (instance.tmuxMode) {
  const session = tmuxManager.getSession(id)
  if (session) {
    const remaining = session.feedData(data)
    if (remaining) {
      // %exit encountered — remaining data is shell output
      instance.tmuxMode = false
      window.webContents.send(IPC.PTY_OUTPUT, id, remaining)
    }
    return
  }
}
```

**Files modified:** `src/main/pty-manager.ts`, `src/main/tmux/tmux-manager.ts`, `src/main/tmux/tmux-session.ts`, `src/main/tmux/tmux-protocol-parser.ts`

## 6. Bug #5: Scrollback Not Loaded on Attach

**Bug description:** When attaching to an existing tmux session with content, the terminal showed blank — no pre-existing scrollback/output was visible.

**How it manifested:** Attaching to a session that had been running commands (e.g., `ls`, `vim`) showed an empty terminal.

**Root cause (initial):** Scrollback was sent via separate IPC events during `initialize()`, but the renderer hadn't mounted TerminalView components yet to receive them.

**Fix (initial):** Scrollback data embedded in the `TmuxSessionInfo` payload as `scrollback: Record<string, string>`. TerminalView reads it from the store on mount.

```typescript
// tmux-types.ts:
export interface TmuxSessionInfo {
  sessionId: string
  sessionName: string
  triggerPtyId: string
  windows: TmuxWindowInfo[]
  scrollback: Record<string, string> // tmuxPaneId -> scrollback data
}
```

> **Note:** This had multiple cascading root causes discovered over subsequent rounds — see Bug #8, Bug #9, and Bug #7.

## 7. Bug #6: Pane Resize Doesn't Work

**Bug description:** Dragging the split divider between tmux panes did nothing.

**How it manifested:** The divider could be dragged but the panes didn't change size.

**Root cause:** `resize-pane -t %PANE -x AMOUNT` uses absolute sizing, but the code passed a delta (number of columns/rows to change by).

**Fix:** Changed to relative flags: `-R`/`-L` for horizontal, `-D`/`-U` for vertical, with `Math.abs(amount)`.

**File modified:** `src/main/tmux/tmux-session.ts`

```typescript
// BEFORE (broken):
resizePane(paneId: string, direction: 'x' | 'y', amount: number): void {
  this.queue.send(`resize-pane -t ${paneId} -${direction} ${amount}`).catch((err) => {
    logger.warn({ paneId, error: err.message }, 'resize-pane failed')
  })
}

// AFTER (fixed):
resizePane(paneId: string, direction: 'x' | 'y', amount: number): void {
  if (amount === 0) return
  let flag: string
  if (direction === 'x') {
    flag = amount > 0 ? '-R' : '-L'
  } else {
    flag = amount > 0 ? '-D' : '-U'
  }
  this.queue.send(`resize-pane -t ${paneId} ${flag} ${Math.abs(amount)}`).catch((err) => {
    logger.warn({ paneId, error: err.message }, 'resize-pane failed')
  })
}
```

## 8. Bug #7: `\r` Corruption in Protocol Parser

**Bug description:** All parsed data from tmux control mode had trailing `\r` characters, corrupting session names, window IDs, pane IDs, layout strings, and command responses.

**How it manifested:** Logs showed `sessionName: "2\r"` instead of `sessionName: "2"`. Command responses were silently corrupted, causing downstream parsing failures.

**Root cause:** Tmux control mode sends `\r\n` line endings, but the protocol parser only split on `\n`, leaving `\r` at the end of every parsed line.

**Fix:** Strip trailing `\r` in `processLines()` before parsing, and add `.trim()` to session name parsing as an extra safety measure.

**Files modified:** `src/main/tmux/tmux-protocol-parser.ts`, `src/main/tmux/tmux-session.ts`

```typescript
// BEFORE — tmux-protocol-parser.ts processLines():
private processLines(): void {
  let nlIndex: number
  while (!this.exited && (nlIndex = this.buffer.indexOf('\n')) !== -1) {
    const line = this.buffer.substring(0, nlIndex)
    this.buffer = this.buffer.substring(nlIndex + 1)
    this.parseLine(line)
  }
}

// AFTER:
private processLines(): void {
  let nlIndex: number
  while (!this.exited && (nlIndex = this.buffer.indexOf('\n')) !== -1) {
    let line = this.buffer.substring(0, nlIndex)
    this.buffer = this.buffer.substring(nlIndex + 1)
    // Strip trailing \r — tmux control mode sends \r\n line endings
    if (line.endsWith('\r')) {
      line = line.slice(0, -1)
    }
    this.parseLine(line)
  }
}

// tmux-session.ts:
this.sessionName = (sessionLines[0] || 'tmux').trim()
```

**Impact of this bug:** This single issue corrupted:
- Session names (`"2\r"` instead of `"2"`)
- Window IDs, pane IDs, layout strings
- Command responses (including `refresh-client` and `capture-pane`)

## 9. Bug #8: `refresh-client` Comma vs X Separator

**Bug description:** The tmux `refresh-client -C` command was silently failing, leaving the client at default (tiny) size.

**How it manifested:** Same symptoms as Bug #3 — input restricted to first column, scrollback capture returned empty. This was the deeper root cause behind both issues.

**Root cause:** Tmux `refresh-client -C` expects `WxH` format (e.g., `200x50`), NOT `W,H` (e.g., `200,50`). Using a comma instead of `x` made the command silently fail.

**Fix:** Changed all `refresh-client -C` calls from comma to `x` separator. Also moved the initial `refresh-client` to run BEFORE `capture-pane` so panes have proper dimensions when their content is captured.

**File modified:** `src/main/tmux/tmux-session.ts`

```typescript
// BEFORE (broken):
this.queue.send(`refresh-client -C ${cols},${rows}`)
await this.queue.send('refresh-client -C 200,50')

// AFTER (fixed):
this.queue.send(`refresh-client -C ${cols}x${rows}`)
await this.queue.send('refresh-client -C 200x50')

// Also reordered in initialize():
// Set client size BEFORE capturing scrollback so panes have proper dimensions
await this.queue.send('refresh-client -C 200x50').catch(() => {})

// Capture scrollback for each pane (now after refresh-client)
const scrollback: Record<string, string> = {}
for (const win of this.windows) {
  for (const pane of win.panes) {
    const scrollbackLines = await this.queue.send(
      `capture-pane -t ${pane.paneId} -p -e -S -`
    )
    // ...
  }
}
```

**Impact:** This single character change (`","` -> `"x"`) fixed two major symptoms:
1. Input no longer restricted to first column
2. Scrollback capture now returns actual content

## 10. Bug #9: Scrollback Consumed by React StrictMode

**Bug description:** Scrollback data was loaded correctly from tmux but appeared blank in the terminal.

**How it manifested:** Logs confirmed scrollback data was present in the store, but terminals showed no pre-existing content.

**Root cause:** `consumeTmuxScrollback()` was destructive — it removed scrollback data from the Zustand store after reading it. With React StrictMode's double-mount in dev:
1. First mount -> `consumeTmuxScrollback(tmuxPaneId)` — removes scrollback from store, writes to terminal
2. StrictMode unmount -> terminal disposed
3. Second mount -> `consumeTmuxScrollback(tmuxPaneId)` — scrollback already gone, nothing to write

**Fix:** Changed to non-destructive read. Scrollback naturally gets cleaned up when `clearTmuxMode()` resets the store.

**File modified:** `src/renderer/components/Terminal/TerminalView.tsx`

```typescript
// BEFORE (broken):
if (tmuxPaneId) {
  const scrollback = useStore.getState().consumeTmuxScrollback(tmuxPaneId)
  if (scrollback) {
    terminal.write(scrollback)
  }
}

// AFTER (fixed):
if (tmuxPaneId) {
  // Read non-destructively — React StrictMode double-mounts would lose consumed data
  const scrollback = useStore.getState().tmuxScrollback[tmuxPaneId]
  if (scrollback) {
    terminal.write(scrollback)
  }
}
```

## 11. Bug #10: Resize Instability / Layout Change Remounting

**Bug description:** Resizing the window caused visual instability — terminals would flicker, lose state, or show incorrect sizes.

**How it manifested:** Window resize caused xterm instances to be destroyed and recreated, losing cursor position and content.

**Root cause (two sub-issues):**

**Part A:** `%layout-change` notifications replaced the entire SplitNode tree with fresh `paneId` values. React saw completely new component keys, so it unmounted all existing TerminalViews and mounted new ones — destroying xterm instances and losing all state.

**Part B:** Each TerminalView was sending its own pane size as `refresh-client -C`, but `refresh-client` sets the *total client size*, not individual pane sizes. Sending per-pane sizes was incorrect for multi-pane layouts.

**Fix Part A:** Added `mergeLayoutTree()` that preserves existing leaf `paneId` values where `tmuxPaneId` matches, so React keeps existing components.

**File modified:** `src/renderer/App.tsx`

```typescript
/**
 * Merge a new SplitNode tree with an existing one, preserving leaf `paneId` values
 * where tmuxPaneId matches. This prevents React from unmounting/remounting
 * TerminalViews when layout changes only modify ratios.
 */
function mergeLayoutTree(oldNode: SplitNode, newNode: SplitNode): SplitNode {
  if (newNode.type === 'leaf' && oldNode.type === 'leaf') {
    if (newNode.tmuxPaneId === oldNode.tmuxPaneId) {
      // Same tmux pane — keep the existing paneId so React doesn't remount
      return { ...newNode, paneId: oldNode.paneId }
    }
  }
  if (newNode.type === 'split' && oldNode.type === 'split') {
    return {
      ...newNode,
      first: mergeLayoutTree(oldNode.first, newNode.first),
      second: mergeLayoutTree(oldNode.second, newNode.second)
    }
  }
  return newNode
}

// Used in layout-change handler:
const offLayoutChange = window.terminalAPI.onTmuxLayoutChange(
  (windowId: string, newRootNode: SplitNode) => {
    const state = useStore.getState()
    const tabId = state.tmuxWindowToTab[windowId]
    if (tabId) {
      const tab = state.tabs.find((t) => t.id === tabId)
      if (tab) {
        const merged = mergeLayoutTree(tab.root, newRootNode)
        updateTabRoot(tabId, merged)
      }
    }
  }
)
```

**Fix Part B:** Removed per-pane `resizeTmux` calls. Added a single `ResizeObserver` on the `.app__content` container that computes total cols/rows and sends one `refresh-client`.

**File modified:** `src/renderer/App.tsx`, `src/renderer/components/Terminal/TerminalView.tsx`

## 12. Bug #11: Mouse Events Not Propagating to Tmux Panes

**Bug description:** Mouse clicks and scrolls in tmux panes did nothing when running mouse-aware applications (vim, htop).

**How it manifested:** Running `vim` or `htop` in a tmux pane showed no response to mouse clicks, scrolling, or drags.

**Investigation process:**

1. **Initial hypothesis:** Tmux doesn't forward mouse-enable sequences (`\x1b[?1000h`) in `%output` for control mode clients, so xterm.js never enters mouse mode.

2. **Tested by adding diagnostic logging** to `tmux-session.ts handleOutput()` to detect mouse sequences in `%output`.

3. **Discovery:** Tmux DOES forward mouse sequences in `%output`. Confirmed seeing `\x1b[?1000h` from vim/htop in the logs.

4. **Second hypothesis:** xterm.js receives the sequences but the `dimensions` error from WebGL addon prevents mouse mode activation.

5. **First attempted fix:** Force-enable mouse reporting in xterm.js by writing `\x1b[?1000h`, `\x1b[?1002h`, `\x1b[?1006h` directly. **Result:** Mouse mode activated for ALL panes including regular shells, causing garbage escape sequences in non-mouse-aware apps.

6. **Reverted force-enable.** Let tmux's natural `%output` forwarding handle mouse mode activation.

7. **Root cause identified:** xterm.js's `MouseService.getCoords()` fails when `renderService.dimensions` is undefined (caused by WebGL addon initialization timing). Even though xterm.js enters mouse mode, it can't calculate cell coordinates for click events.

**Final fix:** Implemented a manual mouse event handler that bypasses xterm.js's broken `MouseService` entirely.

**File modified:** `src/renderer/hooks/useTerminal.ts`

```typescript
// Manual mouse event handling implementation:

class MouseModeTracker {
  isActive = false
  sgrMode = false  // SGR extended mode (\x1b[?1006h)

  processOutput(data: string): void {
    // Scan for mouse mode enable/disable sequences
    if (data.includes('\x1b[?1000h')) this.isActive = true
    if (data.includes('\x1b[?1000l')) this.isActive = false
    if (data.includes('\x1b[?1006h')) this.sgrMode = true
    if (data.includes('\x1b[?1006l')) this.sgrMode = false
  }

  reset(): void {
    this.isActive = false
    this.sgrMode = false
  }
}

// In attach():
const mouseTracker = mouseTrackerRef.current
let pressedButton = -1

const getCellCoords = (e: MouseEvent) => {
  const rect = container.getBoundingClientRect()
  const x = e.clientX - rect.left
  const y = e.clientY - rect.top
  const cellWidth = rect.width / terminal.cols
  const cellHeight = rect.height / terminal.rows
  return {
    col: Math.floor(x / cellWidth) + 1,  // 1-based
    row: Math.floor(y / cellHeight) + 1   // 1-based
  }
}

const encodeMouseEvent = (tracker, type, button, coords) => {
  if (tracker.sgrMode) {
    // SGR format: \x1b[<button;col;row;M/m
    const b = type === 'release' ? button : button
    const suffix = type === 'release' ? 'm' : 'M'
    return `\x1b[<${b};${coords.col};${coords.row}${suffix}`
  }
  // Legacy format (not shown for brevity)
}

// Event listeners on container (capture phase):
container.addEventListener('mousedown', handleMouseDown, { capture: true })
container.addEventListener('mouseup', handleMouseUp, { capture: true })
container.addEventListener('mousemove', handleMouseMove, { capture: true })
container.addEventListener('wheel', handleWheel, { capture: true })
container.addEventListener('contextmenu', handleContextMenu, { capture: true })
```

## 13. Bug #12: WebGL Addon Dimension Errors

**Bug description:** The WebGL addon caused `dimensions` errors during initialization that broke xterm.js's escape sequence processing.

**How it manifested:** Console errors about undefined dimensions. Mouse mode sequences from tmux were received but xterm.js couldn't act on them.

**Root cause:** Loading WebGL addon synchronously before the first `fit()` call caused the render service dimensions to be undefined. This broke `MouseService.getCoords()` and potentially other escape sequence processing.

**Fix (evolved through multiple iterations):**

1. **First attempt:** Deferred WebGL loading with `requestAnimationFrame` — load after initial fit. Partially worked.
2. **Second attempt:** Disabled WebGL entirely, falling back to canvas renderer. Mouse worked correctly.
3. **Final approach:** Fit first with canvas renderer, then load WebGL after a frame.

**File modified:** `src/renderer/hooks/useTerminal.ts`

```typescript
// BEFORE (broken):
terminal.open(container)
try {
  const webglAddon = new WebglAddon()
  webglAddon.onContextLoss(() => webglAddon.dispose())
  terminal.loadAddon(webglAddon)
} catch {
  // WebGL not available
}
fitAddon.fit()

// AFTER (fixed):
terminal.open(container)

// Fit first with the default canvas renderer
fitAddon.fit()

// Load WebGL addon after a frame so the renderer is fully initialized.
// Loading it synchronously before fit can cause "dimensions" errors
```

## 14. Bug #13: Mouse Button Swap & Duplicate Events

**Bug description:** Left click behaved like right click in vim and vice versa. Additionally, a single click would sometimes trigger visual mode selection in vim.

**How it manifested:** Clicking in vim caused the cursor to jump to the correct position, then immediately enter visual mode selection — as if two clicks happened.

**Root cause:** React StrictMode double-mounts caused duplicate mouse event listeners. ## 17. Bug #16: Pane-Driven Resize System

**Bug description:** Computing tmux client size from pixel measurements in the renderer was fundamentally flawed — it could never exactly match xterm.js's internal FitAddon calculations.

**How it manifested:** Column mismatches, content shifts, and resize instability.

**Root cause:** The renderer tried to compute total cols/rows from pixel dimensions of the content area, subtracting padding and dividers. But this estimation could never match what FitAddon calculated internally.

**Fix:** Complete redesign — pane-driven resize system:

1. Each xterm.js terminal's `FitAddon` fits and determines exact cols/rows from font metrics
2. `TerminalView.handleResize` sends those exact dimensions to main process via `tmuxPaneResized(tmuxPaneId, cols, rows)`
3. `TmuxSession.paneResized()` stores each pane's size, debounces, then walks the layout tree
4. For splits: sums pane cols + dividers (vertical) or pane rows + dividers (horizontal)
5. Sends `refresh-client -C totalColsxtotalRows` to tmux

**New IPC channel added:** `tmux:pane-resized`

**Files modified:** `src/shared/tmux-ipc-channels.ts`, `src/preload/api.ts`, `src/preload/index.ts`, `src/main/ipc-handlers.ts`, `src/main/tmux/tmux-session.ts`, `src/renderer/components/Terminal/TerminalView.tsx`, `src/renderer/App.tsx`

```typescript
// tmux-ipc-channels.ts — added:
PANE_RESIZED: 'tmux:pane-resized',

// preload/api.ts — added:
tmuxPaneResized(tmuxPaneId: string, cols: number, rows: number): void

// TerminalView.tsx handleResize:
if (tmuxPaneIdRef.current) {
  window.terminalAPI.tmuxPaneResized(tmuxPaneIdRef.current, cols, rows)
} else {
  window.terminalAPI.resizePty(ptyIdRef.current, cols, rows)
}

// tmux-session.ts — new methods:
private paneSizes = new Map<string, { cols: number; rows: number }>()
private resizeTimer: ReturnType<typeof setTimeout> | null = null
private lastClientCols = 0
private lastClientRows = 0

paneResized(tmuxPaneId: string, cols: number, rows: number): void {
  this.paneSizes.set(tmuxPaneId, { cols, rows })
  // Debounce: multiple panes report around the same time
  if (this.resizeTimer) clearTimeout(this.resizeTimer)
  this.resizeTimer = setTimeout(() => this.computeAndSendClientSize(), 100)
}

private computeAndSendClientSize(): void {
  // Find window containing the last reported pane
  const targetWin = this.windows.find(w => /* ... */)
  const size = this.computeNodeSize(targetWin.rootNode)
  if (size.cols > 0 && size.rows > 0 &&
      (size.cols !== this.lastClientCols || size.rows !== this.lastClientRows)) {
    this.lastClientCols = size.cols
    this.lastClientRows = size.rows
    this.refreshClient(size.cols, size.rows)
  }
}

private computeNodeSize(node: SplitNode): { cols: number; rows: number } {
  if (node.type === 'leaf') {
    const size = this.paneSizes.get(node.tmuxPaneId!)
    return size || { cols: 0, rows: 0 }
  }
  const first = this.computeNodeSize(node.first)
  const second = this.computeNodeSize(node.second)
  if (node.direction === 'vertical') {
    // Side by side: sum cols + 1 for divider, max rows
    return { cols: first.cols + second.cols + 1, rows: Math.max(first.rows, second.rows) }
  } else {
    // Stacked: max cols, sum rows + 1 for divider
    return { cols: Math.max(first.cols, second.cols), rows: first.rows + second.rows + 1 }
  }
}
```

## 18. Bug #17: Resize Oscillation Feedback Loop

**Bug description:** After implementing pane-driven resize, the window entered an infinite resize loop — constantly oscillating between sizes.

**How it manifested:** The terminal content flickered rapidly as sizes changed back and forth.

**Root cause:** Feedback loop: pane resize -> `refresh-client` -> tmux sends `%layout-change` -> renderer updates layout -> FitAddon fits -> pane reports new size -> `refresh-client` -> repeat. **Fix (three layers):**

**Layer 1 — Skip if total unchanged:**
```typescript

    (size.cols !== this.lastClientCols || size.rows !== this.lastClientRows)) {
  // Only send if changed
}
```

```typescript
  this.paneSizes.set(tmuxPaneId, { cols, rows })

  if (Date.now() - this.layoutChangeTime < 300) {
    return
  }
  // ... debounce and compute
}

private handleLayoutChange(windowId: string, layout: string): void {
  this.layoutChangeTime = Date.now()
  // ... process layout change
}
```

**Layer 3 — Increased debounce:**
```typescript

```

## 19. Bug #18: Tab Switch Size Issues

**Bug description:** When switching between tmux tabs (windows), the newly activated tab rendered at the wrong size.

**How it manifested:** Switching tabs showed content at incorrect dimensions — sometimes too narrow, sometimes too wide.

**Root cause (three sub-issues discovered iteratively):**

1. **Wrong window lookup:** `computeAndSendClientSize()` used `this.windows.find(w => w.active)` but the `active` flag was only set during init, not updated when switching tabs.
   - **Fix:** Find the window containing the pane that just reported, not the "active" one.

2. **New panes suppressed:** When switching tabs, tmux sends `%layout-change` which sets the suppression timer. The new tab's panes mount and report sizes, but they're suppressed by the 300ms window.
   - **Fix:** Only suppress panes that already reported (re-reporting due to feedback), not new panes reporting for the first time.

3. **Same total size skipped:** When switching tabs, the total client size might be the same, so `computeAndSendClientSize` skips sending `refresh-client`. But tmux needs a refresh to re-layout the new window's panes.
   - **Fix:** When a new pane reports (tab switch), force the refresh even if the total size hasn't changed.

**File modified:** `src/main/tmux/tmux-session.ts`

```typescript
private forceNextResize = false

paneResized(tmuxPaneId: string, cols: number, rows: number): void {
  const isNewPane = !this.paneSizes.has(tmuxPaneId)
  this.paneSizes.set(tmuxPaneId, { cols, rows })

  // New panes (e.g. from tab switch) force a resize
  if (isNewPane) {
    this.forceNextResize = true
  }

  // Suppress re-reports for 300ms after %layout-change (feedback loop break)
  // but NOT new panes — they need to report to set initial size
  if (!isNewPane && Date.now() - this.layoutChangeTime < 300) {
    return
  }

  // ... debounce and compute
}

private computeAndSendClientSize(): void {
  const size = this.computeNodeSize(targetWin.rootNode)
  const force = this.forceNextResize
  this.forceNextResize = false
  if (size.cols > 0 && size.rows > 0 &&
      (force || size.cols !== this.lastClientCols || size.rows !== this.lastClientRows)) {
    this.lastClientCols = size.cols
    this.lastClientRows = size.rows
    this.refreshClient(size.cols, size.rows)
  }
}
```

## 22. Protocol Parser Architecture

**File:** `src/main/tmux/tmux-protocol-parser.ts`

The parser is a line-buffered state machine that processes tmux control mode protocol:

- **Normal state:** Parses line-by-line, identifying notification types:
  - `%begin SEQ FLAGS CMD_NUMBER` — Start of command response
  - `%end SEQ FLAGS CMD_NUMBER` — End of command response
  - `%error SEQ FLAGS CMD_NUMBER` — Command error
  - `%output %PANE DATA` — Pane output (terminal data)
  - `%window-add @ID` — Window created
  - `%window-close @ID` — Window destroyed
  - `%window-renamed @ID NAME` — Window renamed
  - `%layout-change @ID LAYOUT` — Layout changed
  - `%exit [REASON]` — Session ended

- **In-block state (between `%begin` and `%end`):** Collects response lines for the current command.

- **Exited state:** After `%exit`, returns all remaining data as raw shell output.

**Key design decisions:**
- Line buffering with `\n` delimiter, with `\r` stripping for tmux's `\r\n` endings
- `%output` data passes through octal escape decoding (`tmux-escape.ts`)
- Parser emits notifications via callback to TmuxSession
- `%begin/%end` responses correlated via sequence numbers in TmuxCommandQueue

**Scrollback line joining:**
```typescript
// Lines must be joined with \r\n, not \n, for xterm.js:
const data = scrollbackLines.join('\r\n')
if (data.trim()) {
  scrollback[pane.paneId] = data + '\r\n'
}
```

## 23. AbortController for StrictMode Cleanup

**File:** `src/renderer/hooks/useTerminal.ts`

React StrictMode double-mounts caused duplicate event listeners, leading to:
- Duplicate mouse events (visual mode activation on single click in vim)
- Stale listener references from first mount persisting after unmount

**Solution:** `AbortController` pattern for automatic cleanup:

```typescript
const mouseAbortRef = useRef<AbortController | null>(null)

// In attach():
const mouseAbort = new AbortController()
mouseAbortRef.current?.abort() // Clean up any previous listeners
mouseAbortRef.current = mouseAbort

// All event listeners use the signal:
const listenerOpts = { capture: true, signal: mouseAbort.signal }
container.addEventListener('mousedown', handleMouseDown, listenerOpts)
container.addEventListener('mouseup', handleMouseUp, listenerOpts)
container.addEventListener('mousemove', handleMouseMove, listenerOpts)
container.addEventListener('wheel', handleWheel, listenerOpts)
container.addEventListener('contextmenu', handleContextMenu, listenerOpts)

mouseAbortRef.current?.abort()
```

This ensures that when StrictMode's first mount is cleaned up, all its event listeners are removed before the second mount adds new ones.
